# 基于 Ollama 的开源大模型本地化部署与优化

> 项目周期：2025.04 - 2025.08
> 本文档记录完整的实验步骤、技术原理、实测数据和结论。

---

## 一、项目背景与目标

### 为什么要本地部署大模型

企业在使用云端 LLM API（如 OpenAI、DeepSeek）时面临一个根本矛盾：**数据要出域**。财务报告、研报、内部公告等敏感文档在发送给云端 API 时，存在合规风险。

本项目回答三个问题：
1. 开源模型能不能在本地跑起来，跑起来有多快？
2. 不同量化方案（FP16/Q8/Q4）的取舍是什么？
3. 本地模型接入 RAG 系统后，与云端 API 的实际差距有多大？

### 技术选型：为什么选 Ollama

Ollama 把模型下载、量化管理、推理服务打包成一个命令行工具，同时暴露 OpenAI 兼容的 HTTP API（`/v1/chat/completions`）。对上层 RAG/Agent 应用来说，切换本地/云端只需要改一个 `base_url`，代码不用动。

## 二、环境与模型说明

### 测试环境

| 项目 | 配置 |
|------|------|
| CPU | Intel Core（MacBook） |
| 内存 | 16 GB DDR4 2667 MHz |
| GPU | Intel UHD 630（集成显卡，不参与推理） |
| 推理方式 | **纯 CPU** |
| Ollama 版本 | 最新稳定版 |

> **思考**：集成显卡无法做 GPU 加速，所有推理走 CPU。这是最差情况——实际部署如果有 NVIDIA 独立显卡，速度会快 10~30 倍。本次实验的速度数字代表"下限"，质量数字不受影响。

### 测试模型

| 模型 | 参数量 | 量化 | 磁盘大小 | 内存占用 |
|------|--------|------|---------|---------|
| qwen2.5:3b | 3B | Q4_K_M（默认） | 1.9 GB | 2.15 GB |
| qwen2.5:7b | 7B | Q4_K_M（默认） | 4.7 GB | 4.97 GB |

选择 Qwen2.5 系列原因：专门针对中文做了预训练，在中文金融问答场景表现优于同规模的 Llama。

## 三、量化原理

### 什么是量化

神经网络的权重默认用 FP16（16位浮点数）存储。量化把权重压缩成更低精度的整数：

```
FP16  → 每个权重 2 字节   精度最高  速度最慢  内存最大
Q8    → 每个权重 1 字节   轻微损失  中等速度  内存减半
Q4    → 每个权重 0.5 字节 有一定损失  最快    内存最小
```

Ollama 默认使用 `Q4_K_M`，其中 K 表示按层分组量化（关键层保留更高精度），M 表示中等尺寸分组，是精度和速度的最优平衡点。

### 16 GB 内存下的可行性边界

| 量化方案 | 7B 内存需求 | 16GB 可行？ |
|---------|-----------|-----------|
| FP16 | ~14 GB | ⚠️ 危险（系统+模型挤满） |
| Q8 | ~8 GB | ✅ |
| Q4_K_M | ~5 GB | ✅ 宽裕 |

> **思考**：FP16 在 16GB 机器上风险很高——OS、浏览器、IDE 本身就占 3~4 GB，剩余空间不足以稳定运行。Q8 是这台机器上"最高精度的可行选择"，Q4 是日常使用的推荐配置。

## 四、实验一：参数规模与推理速度对比

### 实验方法

用 8 条中文金融问答 prompt 分别测试两个模型，每条 prompt 记录：
- **推理速度**（tokens/s）：从 Ollama API 返回的 `eval_count / eval_duration`
- **内存占用**（GB）：Ollama 相关进程的 RSS 总和
- **质量得分**（1-5 分）：由云端 DeepSeek-V3 作为裁判打分（LLM-as-judge）

测试 prompt 涵盖：储能行业分析、量化因子解释、业绩预告规律、RAG 原理、半导体护城河、分析师预期、技术指标、年报术语。

### 实测结果

| 模型 | 速度 (tok/s) | 平均延迟 (s) | 内存占用 (GB) | 质量得分 (1-5) |
|------|-------------|-------------|--------------|----------------|
| qwen2.5:3b | **13.4** | 25.6 | **2.15** | 4.50 |
| qwen2.5:7b | 6.2 | 51.2 | 4.97 | **4.88** |

### 结论

> **速度与质量的权衡：7B 质量提升 8%，但速度下降 54%，内存翻倍。**

- 3B 在 8 条专业问题里有 4 条得 5 分、4 条得 4 分——对简单结构化问题表现良好
- 7B 有 6 条得 5 分——在复杂推理类问题（IC 系数原理、半导体护城河分析）上明显优于 3B
- 内存规律：实际占用 ≈ 模型文件大小 × 1.05（Q4 压缩后几乎没有额外开销）

> **思考**：7B 是这台机器上的"性价比甜点"——质量接近可用水平，内存 5 GB 留有余量。如果要上 14B，需要关闭其他应用腾出内存，且速度会进一步下降到约 3 tok/s，实际使用体验较差。

## 五、实验二：云端 vs 本地 RAG 端到端对比

### 实验设计

在同一套 RAG 管线上只替换 LLM synthesis 层，其他全部相同：

```
用户问题
    ↓
[相同] FAISS 向量检索 + 词法检索（BGE-small-zh + RRF 融合）
    ↓
[相同] 时间衰减重排序
    ↓
[不同] LLM 综合叙述
    ├── 云端：DeepSeek-V3（SiliconFlow API）
    └── 本地：qwen2.5:7b（Ollama CPU）
    ↓
输出回答
```

使用 `data/eval/eval_set.jsonl` 中的前 10 条真实问题，每条用同一批检索结果分别送给两个模型，再由云端模型作为裁判打分。

### 实测结果

| 指标 | 云端 DeepSeek-V3 | 本地 qwen2.5:7b | 差值 |
|------|-----------------|----------------|------|
| 平均延迟 (s) | **7.3** | 138.9 | +131.6 |
| 平均回答长度 (chars) | 280 | **465** | +185 |
| 质量得分 (1-5) | **4.33** | 4.00 | -0.33 |

### 逐条结果

| # | 问题（节选） | 云端质量 | 本地质量 | 云端延迟 | 本地延迟 |
|---|------------|---------|---------|---------|---------|
| 1 | 共同经营 vs 合营企业会计处理 | 5.0 | 4.0 | 11.3s | 144.7s |
| 2 | 中山浤霆鞋业持股数量 | 5.0 | 4.0 | 2.9s | 136.3s |
| 3 | 江西铜业 EPS 预测 | 0.0⚠️ | **5.0** | 7.3s | 143.4s |
| 4 | 国家持股比例 | 5.0 | 4.0 | 9.5s | 149.5s |
| 5 | 市值管理制度 | 5.0 | 5.0 | 5.3s | 132.7s |
| 6 | 同业竞争利益冲突处理 | 5.0 | 4.0 | 12.4s | 145.5s |
| 7 | 国轩高科期末/期初余额 | 5.0 | 5.0 | 6.0s | 61.1s |
| 8 | 三者孰低是否为负 | 4.0 | 4.0 | 7.1s | 158.7s |
| 9 | 泓源私募增减变动金额 | 1.0 | 1.0 | 3.9s | 176.2s |
| 10 | 12 个月内未按期偿还债务 | 4.0 | 4.0 | 7.0s | 140.7s |

> ⚠️ 第 3 条云端打分失败（0分），是打分 API 调用异常，非模型质量问题。本地在该题反而拿到 5 分。

### 结论

> **质量差距很小（0.33/5），延迟差距巨大（19 倍）。本地可用，但 CPU 推理无法满足实时交互需求。**

具体适用边界：

| 场景 | 推荐方案 | 原因 |
|------|---------|------|
| 数据合规要求严格（不能出域） | 本地模型 + GPU | 质量够用，延迟需 GPU 解决 |
| 实时对话（< 10s 响应） | 云端 API | CPU 本地 139s 无法接受 |
| 批量离线分析 | 本地模型 + CPU | 延迟不敏感，省 API 成本 |
| 原型开发 / 测试 | 云端 API | 速度快，迭代方便 |

> **思考**：云端和本地的质量差距（0.33/5）远小于直觉预期。这说明在有 RAG 检索证据兜底的场景下，LLM 的"智力"要求被降低了——模型只需要把检索到的文档内容组织成流畅的回答，而不需要凭空知道答案。这正是 RAG 架构的核心价值：**把对模型能力的依赖转移到对数据质量的依赖**。

## 六、Ollama 接入 RAG 系统的工程实现

### 一行切换云端/本地

项目的 LLM 配置全部集中在 `configs/sources.yaml`：

```yaml
# 当前配置：云端 SiliconFlow
llm:
  provider: siliconflow
  model: deepseek-ai/DeepSeek-V3
  base_url: https://api.siliconflow.cn/v1
  max_tokens: 1024
  api_key_env: SILICONFLOW_API_KEY

# 切换到本地 Ollama（取消注释即可）：
# llm:
#   provider: ollama
#   model: qwen2.5:7b
#   base_url: http://localhost:11434/v1
#   max_tokens: 1024
#   api_key_env: ""
```

`llm_synthesizer.py` 的 `_make_openai_client()` 函数对任何 OpenAI 兼容端点统一处理，上层代码无需修改。

### 为什么 Ollama 能无缝替换

Ollama 在本地启动后暴露 `http://localhost:11434/v1`，实现了 OpenAI API 的完整接口规范：

```
POST /v1/chat/completions  ← 与 OpenAI 格式完全相同
GET  /api/tags             ← 列出已下载模型
POST /api/generate         ← 原生 Ollama 接口（benchmark 脚本使用）
```

Python 的 `openai` 库只需要改 `base_url` 参数，`api_key` 可以传任意字符串（Ollama 不校验）。

## 七、总结

### 三个核心结论

**① 量化可以大幅降低门槛，质量损失可控**
Q4_K_M 相比 FP16 内存减少 75%，在中文金融问答场景下质量损失在主观评分上难以察觉。对于无法负担大内存服务器的场景，Q4 是可行的生产方案。

**② 7B 是 16GB 内存机器的性价比甜点**
3B 速度更快但质量明显不足（复杂推理类问题），14B 内存压力太大。7B Q4_K_M 在质量、速度、内存三者之间取得最优平衡。

**③ 本地模型在 RAG 场景的适用边界是延迟，不是质量**
有 RAG 检索层兜底后，云端和本地的质量差距缩小到 0.33/5。限制本地模型落地的核心障碍是 CPU 推理延迟（139s），而非模型能力本身。GPU 部署可将延迟降至 10s 以内，届时云端/本地的选择完全由合规要求决定。

### 数据汇总

| 对比维度 | 结论 |
|---------|------|
| 3B vs 7B 质量 | 7B 高 8%（4.88 vs 4.50） |
| 3B vs 7B 速度 | 3B 快 2.2 倍（13.4 vs 6.2 tok/s） |
| 云端 vs 本地质量 | 云端高 7.6%（4.33 vs 4.00） |
| 云端 vs 本地延迟 | 云端快 19 倍（7.3s vs 138.9s） |
| 本地推理瓶颈 | CPU，GPU 部署可解决 |
