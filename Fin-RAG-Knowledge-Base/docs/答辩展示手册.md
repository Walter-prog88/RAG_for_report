## 二、第一段：问题背景（2分钟）

> "传统投资研究面临两个痛点。**第一，信息过载**——每天有几百份研报、公告、政策文件，人工根本消化不过来。**第二，信息孤岛**——量化因子和文本信息分开存储，无法联合决策。"

> "大语言模型让我们有机会把两个问题一起解决。但直接用 ChatGPT 做投研有本质缺陷：它看不到你的私有数据，回答没有来源引用，也无法使用本地的量化信号。"

> "所以我做了这个项目：**基于 RAG 的 A 股投研知识库**，把文本检索、量化因子、分析师共识三路信息统一接入一个 Agent，输出有来源引用的 Buy/Watch/Avoid 结论。"

## 四、第三段：RAG 技术实现（4分钟）

### 数据源与清洗

> "知识库四类数据来源："

| 类型 | 规模 | 来源 |
|------|------|------|
| 研报 | 2,218 个 | Tushare 546,466条结构化研报 |
| 交易所公告 | 385 个 | 上交所/深交所 API |
| 新闻 | 409 个 | 财经新闻爬虫 |
| 政策文件 | 19 个 | gov.cn、发改委、证监会、工信部 |

> "所有文档统一写成带 YAML frontmatter 的 Markdown，头部字段：标题、来源、股票代码、发布时间、正文 SHA-256 hash。清洗做了三件事：格式标准化、空行归并保留段落边界、SHA-256 去重。英文年报整体排除——它们和中文年报内容完全重叠，之前占了 6,826 个 chunk，约 11% 索引空间。"

### 切片策略

> "切片用 RecursiveCharacterTextSplitter，分隔符优先级：双换行→单换行→句号→分号，尽量在自然边界切割。chunk_size=900 字符，overlap=150（16% 重叠保证上下文连接）。"

> "有一个单文件 cap 设计：一份300页年报不限制会生成 1000+ chunk，会挤占小公司公告的位置。策略是：前 25% 原样保留（摘要和核心亮点），剩余 75% 均匀采样，总量 cap 到 80。"

### 三层检索管线

```
query（可选 HyDE 改写）
  │
  ├── FAISS IndexFlatIP 召回 top-30
  │
  ├── Cross-Encoder (bge-reranker-base) 精排
  │       30个 (query, chunk) pair → NLI logit → sigmoid
  │
  └── 时间衰减混合（最终排序）
          blended = score × (0.6 + 0.4 × recency)
          研报/公告 半衰期 180天  |  政策文件 半衰期 730天
          → top-k
```

> "同时并行一路词法检索，查询词自动扩展——输入'光模块'扩展为'800G/CPO/1.6T'。两路结果用倒数排名融合（RRF）合并，对股票代码精确匹配特别重要。"

### 5.1 研究过程：从一个问题开始

> "我问自己：这个 RAG 系统的检索质量，我怎么知道它是好的？"

> "回看代码，我只有 4 个 smoke test（`scripts/evaluate_retrieval.py`），逻辑是：top-5 结果里有没有包含预期关键词，有就 PASS。这只是个崩溃检测，不是质量衡量。"

> "于是我开始研究：**Recall@k 是什么，在 RAG 系统里怎么用？**"

> "一个关键的概念澄清：召回率是一个**测量指标**，不是一个操作步骤。我之前把它和 Reranking（精排）混淆了——精排是让 top-30 变成更好的 top-5，召回率是事后量这个 top-5 里有多少是真正相关的。**Reranking 是为了提升 Recall@k，Recall@k 是用来测量 Reranking 有没有效果。**"

> "再往下追：要算 Recall@k，需要 ground truth——每个问题的正确答案是哪些文档。我没有。这就是根本缺口。"

> "我还注意到一个认识论问题：合成评估集存在循环逻辑——问题从 chunk 里生成，答案自然指向该 chunk。它不追求绝对准确，它的价值是提供一把**固定的尺子**，让每次改动都有数字可以比较。"

---

### 5.2 问题一：没有评估集 → 无法量化任何改进

**说：**
> "没有 ground truth，我无法知道 Cross-Encoder 精排是否真的有效，无法知道时间衰减参数设得合不合理，无法比较任何两个版本的系统。这是所有问题里最根本的。"

**实现的解法（`scripts/build_eval_set.py` + `scripts/run_eval.py`）：**
> "我写了两个脚本。第一个脚本从 3,031 个文档里随机采样 60 个 chunk，对每个 chunk 调 LLM 生成一个问题，存成 `data/eval/eval_set.jsonl`——这就是带 ground truth 的评估集，60 个问题，每个问题知道答案应该在哪个文档里。"

> "第二个脚本加载评估集，对每个问题跑检索，计算三个指标：**Recall@5**（top-5 里有多少正确文档被找到）、**MRR**（正确文档平均排在第几位的倒数）、**NDCG@5**（位置加权的召回，排第一比排第五得分高）。现在每次改动参数，跑一遍就知道系统是变好了还是变差了。"

```bash
# 生成评估集（约需 5 分钟，调用 LLM 60 次）
python scripts/build_eval_set.py --n-chunks 60

# 运行评估，自动对比 baseline 和 HyDE
python scripts/run_eval.py
```

### 5.3 问题二：查询改写缺失 → 输入语言和文档语言不匹配

**说：**
> "发现评估体系的缺口之后，我回头看检索管线，找到了第二个问题。"

> "用户输入'宁德时代储能前景如何'，这是问句口语，直接 embed 后去检索，但知识库里的 chunk 写的是'储能业务收入同比增长78%，海外市场占比42%'——同一件事，但语言风格完全不同，向量空间里距离不近，可能找不到。这叫查询语言和文档语言的语义偏移。"

**实现的解法：HyDE（`src/agent/llm_synthesizer.py` + `src/indexer/retriever.py`）**
> "解法是 HyDE（Hypothetical Document Embeddings）。不直接 embed 用户问题，而是先让 LLM 生成一段'假设这个问题的回答可能是什么样的文档内容'，再用这段假设文档去做向量检索。假设文档的语言风格和真实文档接近，向量距离更近，召回效果更好。"

> "实现只改了两处：`llm_synthesizer.py` 加了 `hyde_rewrite()` 函数，`retriever.py` 的 `retrieve()` 加了 `use_hyde=False` 参数，默认关闭，按需开启，不影响现有系统。"

> "现在有了评估集，可以直接运行 `run_eval.py` 对比有没有 HyDE 的 Recall@5 差异——这正是评估集的价值：改动有没有用，数字说话。"

---

### 5.4 问题三：字符截断切片 → chunk 语义不完整

**说：**
> "第三个问题在更底层：数据进入 FAISS 之前的切割方式。"

> "现在按字符数截断，遇到长句会在句子中间切断："

```
chunk_A 末尾："宁德时代储能业务在海外市场的竞争优势主要来自..."
chunk_B 开头："...技术壁垒和成本领先，2024年海外收入占比42%"
```

> "两个 chunk 各自被召回，都是半句话，LLM 拿到的是残缺信息。"

**实现的解法：语义切片（`src/indexer/build_vectorstore.py`）**
> "解法是语义切割：对文档内的相邻句子分别 embed，计算余弦相似度，**相似度突然下降的地方**才是自然的语义边界，在那里才切割。这样每个 chunk 内部语义完整，不会出现半句话。"

> "实现为可配置的开关——`configs/sources.yaml` 里 `chunking.strategy: semantic`——默认仍是 `recursive`，不触发索引重建。需要重建时切换开关，重跑 `scripts/build_index.py` 即可。语义切片和字符切片的效果差异，同样可以用 `run_eval.py` 的 Recall@5 数字来衡量。"

---

### 5.5 三个问题与管线位置的对应关系

> "这三个问题分属管线的不同阶段，每个解法都有对应的可量化验证："

```
用户问题
  ↓
[问题二] 查询改写缺失     → 解法：HyDE rewrite
  ↓
[问题三] chunk 语义不完整  → 解法：SemanticChunker（待重建索引验证）
  ↓
FAISS + Cross-Encoder + 时间衰减
  ↓
[问题一] 无评估体系        → 解法：build_eval_set + run_eval
                              Recall@5 / MRR / NDCG@5
```

> "没有问题一的解法，问题二和三的效果是盲猜。三个修复形成闭环：**先有尺子，再改系统，再量改动效果**。"

## 七、Q&A 预案

### Q：和直接问 ChatGPT 有什么区别？

> "三个根本区别：**私有数据**（ChatGPT 看不到本地 3,031 个文档）、**来源引用**（每个结论都有具体文档出处）、**结构化信号**（融合了 22 个量化因子和分析师一致预期）。"

### Q：召回率怎么衡量的？

> "之前只有 smoke test——Hit@5 二值判断。我识别出这不够，实现了一套完整的评估体系：用 LLM 从知识库里采样生成 60 个带 ground truth 的问答对，然后计算 Recall@5、MRR、NDCG@5。这样每次改动都有数字对比。"

### Q：HyDE 真的有效吗，数字是多少？

> "我实现了评估脚本，`run_eval.py` 会自动跑 baseline 和 HyDE 的对比。具体数字取决于实际运行，典型场景下 HyDE 对开放式问题的 Recall@5 提升在 3~10 个百分点，对精确查询（股票代码搜索）提升不明显甚至有干扰，所以默认是关闭的。"

### Q：语义切片为什么还没有用？

> "语义切片需要重建整个 FAISS 索引——3,031 个文档的句子级 embed，CPU 上约需 1~2 小时。代码已经实现，`configs/sources.yaml` 里改一行 `strategy: semantic` 就能触发。没有立即重建是因为当前索引是工作的，重建前应该用评估集确认参数合理。这是正确的工程顺序：先有尺子，再改系统。"

### Q：评估集是 LLM 生成的，有循环逻辑的问题吗？

> "有，我知道这个局限性。从 chunk 生成的问题天然偏向该 chunk 的词汇，导致评估集比真实用户查询更容易。它的价值不在于绝对准确，而在于提供**固定的比较基准**——同一把尺子量改动前后，相对变化是可信的。更严格的做法是收集真实用户查询并人工标注，那是生产系统上线后才有条件做的事。"

### Q：为什么 Agent 不让 LLM 自己决定调哪些工具？

> "金融场景里这样做很危险——LLM 可能跳过关键的风险检查，或者自己决定不看某类文档。我的设计是固定 workflow：所有工具都调，LLM 只负责最后的叙述，不做流程控制。这样输出是可预期的、可审计的。"

### Q：数据来源有版权问题吗？

> "量化面板来自 Tushare Pro 付费接口，有授权。爬虫采集的是政府和交易所的公开信息。研报原文没有入库，只用了 Tushare 提供的结构化字段（评级、目标价、EPS 预测）。"
